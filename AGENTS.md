# AGENTS.md

## 项目定位

本项目是一个基于 C 语言实现的机箱管理程序，运行于 Linux 环境，通过 UDP 与 Web 网管软件通信，通过串口与机箱和 CHMC 模块通信。

代理在处理本仓库时，应以当前既定分工为准，保持“协议定义 / 公共配置 / 业务函数 / 主流程调度 / 串口配置”这五类职责稳定，不随意打乱文件边界。

## 模块分工

### 1. 协议结构定义

- `chassis_structs_chmc.h`
  - 定义机箱与 CHMC 模块通信协议使用的结构体。
  - 只放置协议结构、字段类型、与协议直接相关的必要声明。
  - 不放业务处理、线程控制、串口配置、网络收发实现。

- `chassis_structs_web.h`
  - 定义机箱与 Web 网管软件通信协议使用的结构体。
  - 只放置 Web 通信协议对应的数据结构和必要声明。
  - 不放业务逻辑或运行流程控制代码。

### 2. 公共头文件与配置

- `chassis_management.h`
  - 引用程序所需标准库头文件。
  - 引用 `chassis_structs_chmc.h` 与 `chassis_structs_web.h`。
  - 定义项目通用宏、参数配置、枚举、全局变量声明、函数声明等公共接口。
  - 作为跨文件共享信息的统一出口。

### 3. 业务功能实现

- `chassis_function.c`
  - 实现程序业务功能相关函数。
  - 当前适合承载日志输出、校验和计算、阈值文件读写、健康状态判断、报文组包、UDP 指令解析等逻辑。
  - 重点是“处理数据并生成业务结果”，不负责程序整体启动与线程生命周期管理。

### 4. 启动、初始化与线程调度

- `chassis_management_main.c`
  - 包含主函数 `main`。
  - 负责参数初始化、网络初始化、线程创建、线程启动和主流程调度。
  - 当前主要线程包括 UDP 接收线程、UDP 发送线程、串口接收线程、串口发送线程。
  - 重点是“程序入口和流程控制”，不承载大段协议解析细节。

### 5. 串口配置

- `serial_config.c`
  - 用于串口的配置与初始化。
  - 包括波特率、数据位、校验位、停止位等配置。
  - 只处理串口设备初始化和串口属性设置，不混入业务协议处理。

## 新增需求时的落点规则

新增内容应尽量保持与原有分工一致，按以下类别落到对应文件中。

### 1. 新增 CHMC 协议相关内容

适用内容：
- 新增机箱与 CHMC 模块通信报文结构体
- 新增 CHMC 协议字段
- 调整 CHMC 协议字段顺序、类型、长度

修改位置：
- 优先修改 `chassis_structs_chmc.h`
- 若新增结构体被其他源文件使用，再同步补充 `chassis_management.h` 中需要公开的声明

不要放到：
- `chassis_function.c`
- `chassis_management_main.c`
- `serial_config.c`

### 2. 新增 Web 协议相关内容

适用内容：
- 新增机箱与 Web 网管软件通信报文结构体
- 新增 Web 查询、配置、回包对应字段
- 调整 Web 协议字段定义

修改位置：
- 优先修改 `chassis_structs_web.h`
- 若有跨文件共享声明，再补充到 `chassis_management.h`

不要放到：
- `serial_config.c`
- `chassis_management_main.c` 的线程流程部分

### 3. 新增公共配置、宏和跨文件声明

适用内容：
- 新增 IP、端口、缓冲区长度、状态常量
- 新增全局变量 `extern` 声明
- 新增函数原型声明
- 新增枚举、公共类型定义

修改位置：
- `chassis_management.h`

说明：
- 只要该内容会被多个 `.c` 文件共用，原则上先考虑放到 `chassis_management.h`

### 4. 新增业务处理函数

适用内容：
- 新增协议解析函数
- 新增报文组包函数
- 新增校验、转换、计算、比较逻辑
- 新增阈值处理、状态判定、文件读写辅助逻辑
- 新增对某类 UDP 指令的业务响应处理

修改位置：
- 优先修改 `chassis_function.c`
- 如需对外暴露函数，再同步在 `chassis_management.h` 添加声明

说明：
- 凡是“输入数据，输出业务结果”的逻辑，默认优先放在 `chassis_function.c`

### 5. 新增初始化流程或线程调度逻辑

适用内容：
- 新增初始化函数调用顺序
- 新增线程创建、线程回收、主循环调度
- 新增启动时资源准备
- 新增程序入口前后的流程控制

修改位置：
- 优先修改 `chassis_management_main.c`
- 如需新增公共声明，再同步修改 `chassis_management.h`

说明：
- `chassis_management_main.c` 负责“何时调用”，不负责承载大量“如何处理协议”的细节

### 6. 新增 UDP 收发线程中的流程控制

适用内容：
- 新增接收线程中对报文类型的分发
- 新增发送线程中对回包条件的调度
- 新增线程内状态切换、发送时机控制

修改位置：
- 流程控制部分放在 `chassis_management_main.c`
- 具体解析、组包、判断逻辑放在 `chassis_function.c`

说明：
- 保持“线程里做调度，业务函数里做处理”的边界

### 7. 新增串口配置或串口初始化内容

适用内容：
- 新增串口设备路径配置
- 修改波特率、校验位、停止位、数据位
- 新增串口初始化步骤
- 新增与 `termios` 配置直接相关的函数

修改位置：
- `serial_config.c`

如需跨文件引用：
- 在 `chassis_management.h` 增加函数声明或全局声明

不要放到：
- `chassis_function.c`
- `chassis_management_main.c` 中实现具体串口参数设置细节

### 8. 新增日志接口或公共辅助接口

适用内容：
- 新增通用日志输出函数
- 新增可被多个模块复用的辅助函数接口

修改位置：
- 实现在 `chassis_function.c`
- 声明在 `chassis_management.h`

说明：
- 若函数明显属于串口属性配置，则仍应放在 `serial_config.c`
- 若函数明显属于线程控制，则仍应放在 `chassis_management_main.c`

## 修改原则

- 优先在既有职责边界内修改，不轻易跨文件扩散逻辑。
- 新增内容先判断属于哪一类，再落到对应文件。
- 如某项新增需求同时涉及“结构定义 + 公共声明 + 业务实现”，允许按职责拆分到多个文件，但每个文件只放自己应负责的部分。
- 若发现现有代码与上述分工不完全一致，默认以“逐步收敛到该分工”为目标，而不是继续扩散混写。

## 明确禁止事项

- 不要把协议结构体定义散落到 `.c` 文件中。
- 不要在 `serial_config.c` 中加入 UDP 或业务组包逻辑。
- 不要在 `chassis_management_main.c` 中堆积大量协议解析和业务判断细节。
- 不要在多个 `.c` 文件中重复实现同名公共函数。
- 不要未经确认就主动重构文件职责边界。

## 注释保护规则

- 在进行代码修改时，不要主动删除已有注释内容。
- 只有在“注释对应的代码被一并删除”时，才允许删除对应注释。
- 若注释表述过时，但对应代码仍保留，默认先保留注释，并在给出建议时说明应否修订。

## 审阅确认规则

- 代理在准备修改代码前，应先给出修改建议、修改解释、修改范围和落点文件。
- 必须等待用户完成审阅并明确同意后，才能实际修改代码。
- 如果用户仅要求分析、说明、定位或给建议，则只提供建议，不直接改动文件。
- 若需求明显会影响多个模块，也应先列出将改动的文件，再等待确认。

## 代码入口与建议阅读顺序

建议代理按以下顺序理解项目：

1. 先看 `chassis_management.h`，了解全局配置、公共声明和依赖头文件。
2. 再看 `chassis_structs_chmc.h` 与 `chassis_structs_web.h`，理解两侧通信协议结构。
3. 然后看 `chassis_management_main.c`，明确程序入口、初始化顺序和线程模型。
4. 最后看 `chassis_function.c` 与 `serial_config.c`，分别理解业务处理和串口配置实现。

## 当前协作假设

- 本仓库当前采用“协议定义 / 公共配置 / 业务函数 / 主流程调度 / 串口配置”五部分分工。
- 后续代理应沿用这套结构处理新增需求。
- 如发现旧实现、重复实现或职责漂移，应先说明影响和建议，再等待用户确认后修改。
